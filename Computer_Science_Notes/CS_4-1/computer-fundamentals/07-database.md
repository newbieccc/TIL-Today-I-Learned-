## 1. 데이터베이스 개요

### 1.1 정보사회와 데이터

정보사회에서는 다양한 영역의 활동에 따라 매우 많은 양의 데이터가 발생한다.

| 영역         | 발생하는 데이터 예                                      |
|------------|-------------------------------------------------|
| 항공예약       | 예약자, 출발지, 목적지, 항공사, 탑승시간, 도착시간, 좌석 등            |
| 판매         | 상품코드, 수량, 단가 등                                  |
| SNS 게시물 작성 | 게시물 ID, 사용자 ID, 콘텐츠, 해시태그, 첨부 미디어, 게시일시, 공개범위 등 |

데이터는 단순히 저장되는 것에 그치지 않고, 검색·갱신·관리되어야 의미 있는 정보로 활용될 수 있다.

---

### 1.2 파일

파일(file)이란 보조기억장치에 저장되어 있는 데이터의 집합이다.

- 보조기억장치에 저장되어 있음
- 보관되어 있는 위치와 이름으로 접근하여 사용함
- 특정 프로그램이나 사용 목적에 따라 데이터를 저장하는 기본 단위로 사용됨

---

### 1.3 파일 구조

파일에 저장된 레코드를 검색하기 위해 여러 가지 파일 구조가 사용된다.

| 파일 구조       | 핵심 내용                   | 특징                                    |
|-------------|-------------------------|---------------------------------------|
| 순차적 구조      | 데이터를 순서대로 저장            | 단순하지만 특정 데이터를 찾으려면 앞에서부터 순차적으로 탐색해야 함 |
| 상대적(직접적) 구조 | 키 값을 상대주소로 변환하여 접근      | 키를 이용해 원하는 위치에 직접 접근 가능               |
| 인덱스된 구조     | 키 값과 레코드 주소를 저장한 인덱스 사용 | 인덱스를 통해 레코드를 빠르게 검색 가능                |

#### 순차적 구조

- 레코드가 저장된 순서대로 배치됨
- 예: 사번 순서대로 직원 정보가 저장된 파일
- 특정 레코드를 찾기 위해 처음부터 차례대로 검색해야 할 수 있음

#### 상대적(직접적) 구조

- 키 값을 이용하여 상대주소를 계산함
- 계산된 주소를 통해 원하는 레코드에 직접 접근함
- 예: 이름 또는 사번 같은 키를 주소로 변환하여 해당 위치 검색

#### 인덱스된 구조

- 키 값과 레코드 주소의 쌍을 인덱스로 저장함
- 인덱스를 이용하면 특정 키 값을 갖는 레코드를 빠르게 액세스할 수 있음
- 시험에서는 “키값과 주소쌍을 쉽게 검색할 수 있는 구조”라는 표현이 나오면 인덱스 구조를 떠올리면 된다.

---

### 1.4 파일을 이용한 데이터 관리

파일 기반 데이터 관리는 응용프로그램 단위로 필요한 파일을 각각 소유하고 관리하는 방식이다.

예를 들어 다음과 같이 관리될 수 있다.

| 응용프로그램    | 사용하는 파일 |
|-----------|---------|
| 급여관리 프로그램 | 급여 파일   |
| 인사관리 프로그램 | 인사 파일   |

이 방식에서는 급여 파일과 인사 파일에 사번, 이름, 주민번호, 주소 같은 데이터가 중복 저장될 수 있다.

---

### 1.5 파일 관리 방식의 문제점

파일을 이용한 데이터 관리에는 다음과 같은 문제가 있다.

| 문제점            | 설명                                            |
|----------------|-----------------------------------------------|
| 데이터가 프로그램에 종속됨 | 특정 응용프로그램에 맞추어 파일 구조가 만들어져 다른 프로그램에서 활용하기 어려움 |
| 데이터 중복 발생      | 서로 다른 응용프로그램이 같은 데이터를 각각 저장할 수 있음             |
| 저장공간 낭비        | 동일한 데이터가 여러 파일에 반복 저장됨                        |
| 데이터 모순 발생 가능성  | 한쪽 파일만 수정되고 다른 파일은 수정되지 않으면 데이터 불일치가 발생함      |

예를 들어 인사 파일과 급여 파일에 같은 직원의 주소가 중복 저장되어 있을 때, 인사 파일의 주소만 수정되고 급여 파일은 수정되지 않으면 데이터 모순이 발생할 수 있다.

---

### 1.6 데이터베이스

데이터베이스(database)란 어떤 조직이나 단체를 유지하고 관리하는 데 필수적인 관련 데이터의 집합이다.

- 논리적으로 연관된 하나 이상의 데이터 집합
- 데이터를 고도로 구조화하여 검색·갱신·관리를 효율화한 것
- 데이터베이스 관리 시스템(DBMS)을 통해 데이터를 저장하거나 활용·관리함

#### 데이터베이스 시스템

데이터베이스 시스템은 DBMS를 이용하여 특정 기관의 데이터를 데이터베이스로 구축하고, 이를 바탕으로 설계된 응용프로그램 집합을 이용해 사용자에게 필요한 정보 서비스를 제공하는 시스템이다.

구성 요소는 다음과 같다.

| 구성 요소           | 역할                         |
|-----------------|----------------------------|
| 데이터베이스          | 실제 데이터가 저장되는 곳             |
| DBMS            | 데이터베이스와 응용프로그램 사이에서 데이터 관리 |
| 응용프로그램          | 사용자의 업무 처리를 위해 데이터베이스를 활용  |
| 사용자             | 응용프로그램을 통해 필요한 정보 서비스 이용   |
| 데이터베이스 관리자(DBA) | 데이터베이스의 전반적인 관리 담당         |

---

## 2. 데이터베이스 관리 시스템

### 2.1 DBMS의 개념

데이터베이스 관리 시스템(DBMS, Database Management System)은 응용프로그램과 데이터 간의 독립성을 유지하고, 다수 사용자에 의한 공용을 지원하는 소프트웨어 패키지이다.

DBMS는 다음과 같은 역할을 한다.

- 다양한 응용에 필요한 여러 유형의 개체에 대한 데이터 관리
- 데이터 사이의 상관관계에 대한 정보 관리
- 응용프로그램과 데이터베이스 사이의 중재자 역할
- 사용자나 응용프로그램이 DBMS를 통해서만 데이터베이스에 접근하도록 함

---

### 2.2 파일 이용 방식과 데이터베이스 이용 방식

| 구분      | 파일 이용 방식                   | 데이터베이스 이용 방식          |
|---------|----------------------------|-----------------------|
| 데이터 관리  | 업무별로 개별 파일 관리              | 통합 데이터베이스 관리          |
| 중복 가능성  | 높음                         | 낮음                    |
| 데이터 일관성 | 유지하기 어려움                   | DBMS를 통해 중앙 관리 가능     |
| 데이터 공유  | 제한적                        | 여러 업무에서 공유 가능         |
| 예       | 인사 파일, 급여 파일, 재고 파일을 각각 관리 | 인사·급여·구매 업무가 통합 DB 사용 |

데이터베이스 이용 방식에서는 여러 업무가 통합 DB를 사용하므로 데이터 중복을 줄이고 일관성을 높일 수 있다.

---

### 2.3 DBMS의 역할

DBMS의 주요 역할은 다음과 같다.

| 역할              | 설명                                |
|-----------------|-----------------------------------|
| 빠른 데이터 저장·검색·갱신 | 데이터를 효율적으로 저장하고 필요한 정보를 빠르게 검색·수정 |
| 데이터 독립성 유지      | 응용프로그램과 데이터 구조의 의존성을 줄임           |
| 데이터 공유 지원       | 여러 사용자가 데이터를 공동으로 이용 가능           |
| 동시실행 제어         | 다수 사용자가 동시에 접근해도 데이터의 정확성 유지      |
| 데이터 저장 및 회복     | 데이터를 안전하게 저장하고 장애 발생 시 회복 가능      |
| 질의언어 및 개발환경 제공  | SQL, 질의처리 기능, 응용프로그램 개발환경 등 제공    |
| 데이터 보안          | 정당한 사용자만 접근할 수 있도록 보안 유지          |

---

### 2.4 DBMS의 필수 기능

DBMS의 필수 기능은 크게 세 가지로 구분된다.

| 기능     | 핵심 내용                         |
|--------|-------------------------------|
| 데이터 정의 | 데이터베이스 구조를 정의하는 기능            |
| 데이터 조작 | 데이터베이스에 접근하여 검색·삽입·수정·삭제하는 기능 |
| 데이터 제어 | 데이터의 정확성과 안전성을 유지하는 기능        |

---

#### 2.4.1 데이터 정의 기능

데이터 정의 기능은 다양한 응용프로그램을 지원하기 위한 데이터베이스 구조를 정의하는 기능이다.

주요 내용은 다음과 같다.

- 모든 응용프로그램을 위한 데이터의 논리적 구조 정의
- 데이터베이스를 보조기억장치에 저장하기 위한 물리적 구조 정의
- 데이터의 논리적 구조와 물리적 구조 사이의 사상(mapping) 정의

---

#### 2.4.2 데이터 조작 기능

데이터 조작 기능은 사용자의 요구에 따라 데이터베이스에 체계적으로 접근하고 조작하는 기능이다.

- 검색
- 갱신
- 삽입
- 삭제

이러한 연산은 데이터 언어를 통해 지시된다.

데이터 언어의 요건은 다음과 같다.

| 요건       | 설명                              |
|----------|---------------------------------|
| 사용의 용이성  | 사용자가 쉽게 사용할 수 있어야 함             |
| 명확하고 완전함 | 명령의 의미가 명확하고 필요한 기능을 충분히 제공해야 함 |
| 효율성      | 데이터 처리가 효율적으로 이루어져야 함           |

---

#### 2.4.3 데이터 제어 기능

데이터 제어 기능은 공용 목적으로 관리되는 데이터베이스의 내용에 대한 정확성과 안전성을 유지하는 기능이다.

주요 요건은 다음과 같다.

| 요건            | 설명                         |
|---------------|----------------------------|
| 무결성 유지        | 데이터가 정확하고 일관된 상태를 유지하도록 제어 |
| 권한 검사 및 보안 유지 | 정당한 사용자만 데이터에 접근하도록 제어     |
| 병행제어          | 여러 사용자의 동시 접근에도 데이터 정확성 유지 |

---

### 2.5 데이터베이스 언어

데이터베이스 언어는 데이터베이스에 대한 작업을 수행하는 특수 목적의 언어이다.

대표적으로 관계형 데이터베이스 관리 시스템(RDBMS)에서는 SQL(Structured Query Language)을 사용한다.

---

#### 2.5.1 데이터 정의어

데이터 정의어(DDL, Data Definition Language)는 데이터베이스의 스키마를 생성, 변경, 삭제할 목적으로 사용하는 언어이다.

- 데이터베이스 구조를 정의함
- 스키마를 생성·변경·삭제함
- 스키마는 데이터베이스 구조와 제약조건에 대한 명세를 의미함

대표 기능 예시는 다음과 같다.

| 구분 | 의미           |
|----|--------------|
| 생성 | 데이터베이스 구조 생성 |
| 변경 | 기존 구조 변경     |
| 삭제 | 기존 구조 삭제     |

---

#### 2.5.2 데이터 조작어

데이터 조작어(DML, Data Manipulation Language)는 DDL에 의해 정의된 데이터베이스에 대해 데이터를 삽입, 검색, 수정, 삭제하기 위한 언어이다.

| 조작 | 의미         |
|----|------------|
| 삽입 | 새로운 데이터 추가 |
| 검색 | 필요한 데이터 조회 |
| 수정 | 기존 데이터 변경  |
| 삭제 | 기존 데이터 제거  |

---

### 2.6 DDL과 DML 비교

| 구분     | DDL                      | DML                        |
|--------|--------------------------|----------------------------|
| 영어     | Data Definition Language | Data Manipulation Language |
| 목적     | 데이터베이스 구조 정의             | 데이터 처리                     |
| 대상     | 스키마, 구조, 제약조건            | 실제 데이터                     |
| 주요 기능  | 생성, 변경, 삭제               | 삽입, 검색, 수정, 삭제             |
| 시험 포인트 | “스키마 생성·변경·삭제”           | “데이터 삽입·검색·수정·삭제”          |

---

## 3. 데이터베이스 스키마

### 3.1 스키마의 개념

데이터베이스 스키마(schema)란 데이터베이스에서 데이터의 구조, 데이터의 표현 방법, 제약조건 등을 정의한 것이다.

스키마는 데이터베이스를 보는 입장에 따라 여러 단계로 나뉜다.

| 관점          | 설명                    |
|-------------|-----------------------|
| 사용자의 입장     | 사용자가 보는 데이터 구조        |
| 조직 전체의 입장   | 조직 전체에서 보는 통합 데이터 구조  |
| 물리적 저장장치 입장 | 데이터가 실제 저장장치에 저장되는 방식 |

---

### 3.2 3계층 스키마 구조

데이터베이스 스키마는 일반적으로 3계층 구조로 설명된다.

| 스키마    | 관점               | 설명                    |
|--------|------------------|-----------------------|
| 외부 스키마 | 개별 사용자·응용프로그램 입장 | 사용자가 접근하는 데이터베이스 구조   |
| 개념 스키마 | 조직 전체 입장         | 조직 전체의 통합적인 데이터 구조    |
| 내부 스키마 | 저장장치 입장          | 데이터베이스가 물리적으로 저장되는 방법 |

구조를 간단히 표현하면 다음과 같다.

    외부 스키마 1   외부 스키마 2   ...   외부 스키마 n
             ↓          ↓                ↓
                    개념 스키마
                        ↓
                    내부 스키마
                        ↓
                  저장 데이터베이스

---

### 3.3 외부 스키마

외부 스키마는 개별 사용자나 응용프로그램 입장에서 본 데이터베이스 구조이다.

- 사용자가 실제로 접근하는 데이터 구조
- 사용자별, 응용프로그램별로 다를 수 있음
- 시험에서 “개별 사용자나 응용 업무가 접근하는 데이터베이스 구조”라고 나오면 외부 스키마를 의미함

---

### 3.4 개념 스키마

개념 스키마는 조직 전체의 입장에서 본 데이터베이스 구조이다.

- 조직 전체의 통합적인 데이터 구조
- 데이터베이스 전체의 논리적 구조를 나타냄
- 여러 외부 스키마의 공통 기반이 됨

---

### 3.5 내부 스키마

내부 스키마는 물리적 저장장치 입장에서 본 데이터베이스 구조이다.

- 데이터가 실제 저장장치에 어떻게 저장되는지 정의함
- 물리적 저장 구조와 관련됨
- 저장 방법, 접근 경로 등과 관련됨

---

## 4. 데이터베이스 시스템의 장·단점

### 4.1 데이터베이스 시스템의 장점

데이터베이스 시스템을 사용하면 데이터 중복을 줄이고 데이터의 일관성, 무결성, 독립성, 공유성을 높일 수 있다.

| 장점            | 설명                          |
|---------------|-----------------------------|
| 데이터 중복 최소화    | 같은 데이터를 여러 곳에 반복 저장하는 것을 줄임 |
| 데이터 일관성 확보    | 중복을 줄여 데이터 간 불일치를 방지        |
| 저장공간 절감       | 불필요한 중복 저장을 줄여 저장공간을 절약     |
| 데이터 독립성 보장    | 응용프로그램과 데이터 구조 사이의 의존성을 줄임  |
| 데이터 무결성 제공    | 데이터가 정확하고 유효한 상태를 유지하도록 함   |
| 데이터 공유        | 여러 응용프로그램과 사용자가 데이터를 공동 이용  |
| 사용자 인터페이스 표준화 | SQL을 통한 표준 사용자 인터페이스 제공     |

---

### 4.2 데이터 독립성

데이터 독립성은 응용프로그램과 데이터 구조 사이의 의존성을 줄이는 것을 의미한다.

| 종류          | 설명                                       |
|-------------|------------------------------------------|
| 논리적 데이터 독립성 | 데이터의 논리적 구조가 변경되어도 응용프로그램에 미치는 영향을 줄임    |
| 물리적 데이터 독립성 | 데이터의 물리적 저장 방식이 변경되어도 응용프로그램에 미치는 영향을 줄임 |

시험에서는 “데이터 독립성 보장”의 세부 항목으로 논리적 데이터 독립성과 물리적 데이터 독립성을 함께 기억한다.

---

### 4.3 데이터의 일관성 및 무결성

데이터베이스 시스템은 데이터 중복을 최소화하여 데이터의 일관성을 유지한다.

또한 데이터 갱신 시 DBMS의 중앙제어를 통해 유효성 검사를 수행할 수 있으므로 데이터 무결성 유지에 유리하다.

| 개념  | 의미                         |
|-----|----------------------------|
| 일관성 | 같은 데이터가 서로 모순되지 않고 일치하는 상태 |
| 무결성 | 데이터가 정확하고 유효한 상태           |

---

### 4.4 데이터의 공유

데이터베이스는 다양한 응용의 요구사항을 수렴한 통합적 개념 스키마를 통해 데이터를 공유한다.

- 여러 부서나 응용프로그램이 같은 데이터베이스를 활용할 수 있음
- 중복 저장 없이 필요한 정보를 공동 이용할 수 있음
- 다수 사용자의 동시 접근은 DBMS의 병행제어 기능을 통해 관리됨

---

### 4.5 데이터베이스 시스템의 단점

데이터베이스 시스템은 장점이 많지만 다음과 같은 단점도 있다.

| 단점             | 설명                                  |
|----------------|-------------------------------------|
| 운영비 증대         | 고가의 DBMS, 더 높은 CPU 성능, 큰 주기억장치가 필요함 |
| 데이터 처리 방법의 복잡성 | 다양한 응용을 고려한 데이터 구조 설계가 필요함          |
| 전문 인력 필요       | 고급 설계자와 프로그래머가 필요함                  |
| 백업과 회복의 복잡성    | 데이터베이스 전체를 안전하게 백업하고 회복하는 절차가 복잡함   |
| 시스템 취약성        | 중앙 집중적 관리로 인해 시스템 장애 시 영향이 클 수 있음   |

---

## 5. 데이터 웨어하우스

### 5.1 데이터 웨어하우스의 개념

데이터베이스는 현 시점의 정확한 데이터를 저장하는 데 중점을 둔다.

예를 들면 다음과 같다.

- 현재 재직 중인 직원
- 생산 중인 상품
- 재고
- 매출

하지만 데이터는 시간에 따라 변화하므로, 변화의 추이를 관찰하여 기관의 의사결정에 필요한 정보를 도출할 필요가 있다.

이러한 목적을 위해 사용되는 것이 데이터 웨어하우스이다.

---

### 5.2 데이터 웨어하우스란?

데이터 웨어하우스(data warehouse)는 의사 결정을 지원하는 데이터의 집합체이다.

- 기업의 운영 시스템에서 데이터를 추출함
- 마케팅, 판매, 회계 등 운영 시스템의 데이터베이스로부터 데이터를 가져옴
- 데이터를 적절한 형태로 변환하여 저장함
- 분기별, 연도별 현황과 경향 같은 분석 보고서 작성에 활용됨

---

### 5.3 데이터베이스와 데이터 웨어하우스 비교

| 구분     | 데이터베이스                   | 데이터 웨어하우스           |
|--------|--------------------------|---------------------|
| 목적     | 현재 업무 처리                 | 의사 결정 지원            |
| 데이터 성격 | 현재 시점의 정확한 데이터           | 시간에 따라 축적된 분석용 데이터  |
| 활용 예   | 직원, 상품, 재고, 매출의 현재 상태 관리 | 분기별·연도별 현황, 경향 분석   |
| 갱신     | 업무 처리에 따라 계속 갱신          | 주로 읽기 전용, 갱신이 거의 없음 |
| 사용 목적  | 운영 중심                    | 분석 중심               |

---

### 5.4 데이터 웨어하우스의 특징

데이터 웨어하우스의 특징은 네 가지로 정리할 수 있다.

| 특징   | 설명                                   |
|------|--------------------------------------|
| 주제지향 | 의사결정에 필요한 특정 주제별로 구성된 데이터를 저장        |
| 통합   | 다수의 서로 다른 데이터베이스로부터 일관적 형태로 변환되어 통합  |
| 시계열  | 일, 월, 분기, 연도 등 정해 놓은 시점별로 변화된 정보를 저장 |
| 비휘발성 | 갱신이 이루어지지 않는 읽기 전용 데이터베이스            |

---

## 정리하기

1. 데이터를 저장하기 위한 파일의 구조에는 순차적 구조, 상대적 구조, 인덱스 구조 등이 있다.

2. 파일을 이용해 응용프로그램별로 필요한 데이터를 관리할 경우 데이터가 응용프로그램에 종속되고, 데이터 중복이 발생할 수 있다.

3. 데이터베이스는 어떤 조직이나 단체를 유지하고 관리하는 데 필수적인 관련 데이터의 집합이다.

4. 데이터베이스 관리 시스템(DBMS)은 응용프로그램과 데이터 간의 독립성을 유지하고, 다수 사용자에 의한 데이터 공유를 지원하는 소프트웨어 패키지이다.

5. 데이터베이스의 필수 기능에는 데이터 정의 기능, 데이터 조작 기능, 데이터 제어 기능 등이 있다.

6. 데이터베이스에서 데이터의 구조, 데이터의 표현 방법, 제약조건에 대한 명세를 스키마라고 한다.

7. 데이터베이스를 사용하면 데이터 중복의 최소화, 데이터 독립성 제공, 데이터의 일관성과 무결성 제공, 데이터 공유 등의 장점을 얻을 수 있다.

8. 데이터 웨어하우스는 기업과 같은 조직의 의사결정을 지원하는 데이터의 집합체로, 운영 시스템의 데이터베이스로부터 추출되어 적절한 형태로 변환되어 저장된다.

9. 데이터 웨어하우스는 주제지향, 통합, 시계열, 비휘발성이라는 특성을 갖는다.

## 연습문제 정리

### Q1. 다음 중 키를 이용하여 레코드를 쉽게 검색하기 위해 키 값과 레코드 주소를 저장하는 데이터구조를 사용하는 파일 구조는?

1. 인덱스된 파일 구조
2. 상대적 파일 구조
3. 순차적 파일 구조
4. 직접적 파일 구조

정답: 1

해설:
키값과 주소의 쌍을 쉽게 검색할 수 있도록 저장하는 자료구조를 인덱스라고 한다. 인덱스를 이용하면 특정 키값을 갖는 레코드를 빠르게 액세스할 수 있다.

---

### Q2. 한 조직의 정보시스템 내에서 동일한 정보를 응용 업무마다 중복적으로 저장함으로써 야기되는 특성에 해당되는 것은?

1. 데이터 저장공간이 절약된다.
2. 데이터 변경에 따른 데이터 모순성을 발생시킬 가능성이 있다.
3. 데이터의 처리 속도가 높아진다.
4. 정보의 통신이 어려워진다.

정답: 2

해설:
데이터를 응용 업무마다 중복 저장하면 어느 한 응용에서 수정한 데이터가 다른 응용에서 관리하는 데이터와 동시에 수정되지 않을 수 있다. 이 경우 데이터 모순이 발생할 수 있어 정보 서비스의 질을 떨어뜨릴 수 있다.

---

### Q3. 다음 중 데이터베이스의 생성·유지·보수하며 성능의 모니터링 및 개선 등의 데이터베이스 관리 업무를 담당하는 관리자를 의미하는 것은?

1. 응용프로그래머
2. 데이터베이스 관리 시스템(DBMS)
3. 스키마
4. 데이터베이스 관리자(DBA)

정답: 4

해설:
DBA(DataBase Administrator), 즉 데이터베이스 관리자는 데이터베이스의 전반적인 관리 업무를 담당하는 사람을 의미한다.

---

### Q4. 다음 중 데이터베이스 스키마에 대하여 설명한 것으로 적절한 것은?

1. 데이터베이스를 이용하여 동작하는 응용 소프트웨어이다.
2. 데이터베이스 중 특정 응용에 해당되는 데이터를 별도로 분리하여 저장한 것이다.
3. 저장장치의 입장, 조직 전체의 입장, 사용자 또는 응용 업무 입장에서 데이터의 구조, 제약조건 등 데이터베이스의 데이터가 어떻게 구성되는가를 정의한 것이다.
4. 데이터베이스 내의 데이터를 검색하거나 삽입, 수정, 삭제 등의 처리를 하기 위한 프로그래밍 언어이다.

정답: 3

해설:
데이터베이스 스키마는 데이터베이스에서 데이터의 구조, 데이터의 표현 방법, 제약조건에 대한 명세를 기술한 설계도와 같은 것이다. 이러한 설계는 물리적 저장장치 수준, 조직 전체의 수준, 개별 사용자 또는 응용 수준에
대해 이루어진다.

---

### Q5. 다음 중 개별 사용자나 응용 업무가 접근하는 데이터베이스 구조를 정의한 것을 나타내는 것은?

1. 내부 스키마
2. 외부 스키마
3. DML
4. DDL

정답: 2

해설:
외부 스키마는 개별 사용자나 응용프로그램 입장에서 본 데이터베이스 구조를 정의한 것이다. 조직 전체의 입장에서 본 데이터베이스 구조는 개념 스키마, 저장장치 입장에서 본 데이터베이스 저장 방법은 내부 스키마이다.

### Q6. 다음 중 데이터 웨어하우스의 특성을 올바르게 설명한 것은?

1. 조직 운영의 단위 업무와 관련된 데이터의 집합이다.
2. 현 시점의 조직의 업무 데이터를 저장하고 있다.
3. 저장된 데이터는 거래의 발생에 따라 시시각각으로 갱신된다.
4. 일, 월, 분기, 연도 등 정해 놓은 시점별로 변화된 정보를 저장하는 시계열 데이터를 제공한다.

정답: 4

해설:
데이터 웨어하우스는 의사결정에 필요한 특정 주제별 데이터를 저장하는 주제지향성, 여러 데이터베이스로부터 일관적 형태로 통합되는 통합성, 일·월·분기·연도 등 정해 놓은 시점별로 변화된 정보를 저장하는 시계열성,
갱신이 이루어지지 않는 읽기 전용의 비휘발성이라는 특징을 갖는다.

## 시험 직전 체크포인트

### 반드시 구분하기

| 헷갈리는 개념                    | 구분 포인트                         |
|----------------------------|--------------------------------|
| 파일 vs 데이터베이스               | 파일은 프로그램별 개별 관리, 데이터베이스는 통합 관리 |
| DBMS vs 데이터베이스             | DBMS는 관리 소프트웨어, 데이터베이스는 데이터 집합 |
| DDL vs DML                 | DDL은 구조 정의, DML은 데이터 조작        |
| 외부 스키마 vs 개념 스키마 vs 내부 스키마 | 사용자 관점, 조직 전체 관점, 저장장치 관점      |
| 데이터베이스 vs 데이터 웨어하우스        | 현재 업무 처리 중심, 의사결정·분석 중심        |

---

### 암기 키워드

- 파일 구조: 순차적, 상대적, 인덱스
- 파일 관리 문제점: 프로그램 종속, 데이터 중복, 저장공간 낭비, 데이터 모순
- DBMS 역할: 독립성, 공유, 동시실행 제어, 보안, 회복
- DBMS 필수 기능: 데이터 정의, 데이터 조작, 데이터 제어
- 데이터 언어: DDL, DML, SQL
- 스키마 3계층: 외부, 개념, 내부
- 데이터베이스 장점: 중복 최소화, 독립성, 일관성, 무결성, 공유
- 데이터베이스 단점: 비용, 복잡성, 백업·회복, 시스템 취약성
- 데이터 웨어하우스 특징: 주제지향, 통합, 시계열, 비휘발성
