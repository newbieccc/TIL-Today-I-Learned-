## 1. 프로그램 개발 준비

### 1.1 소프트웨어 개발 과정

소프트웨어 개발 과정은 대규모 소프트웨어 개발에 자원, 인력, 시간 등을 투입하여 초기에 예상한 결과물을 구현하기 위한 체계적인 절차이다.

공장에서 제품을 생산하듯 소프트웨어 전체 개발 과정을 하나의 프로세스로 정의한다.

---

### 1.2 소프트웨어 개발 라이프사이클, SDLC

SDLC는 Software Development Life Cycle의 약자이다.

소프트웨어 개발 계획부터 유지보수까지 전체 개발 프로세스를 의미한다.

| 단계 | 이름                    | 핵심 내용                              |
|----|-----------------------|------------------------------------|
| 1  | Planning              | 비용, 기간, 자원 투입 계획 수립                |
| 2  | Analysis              | 소프트웨어 기능·비기능 요구 정의 및 자료 수집         |
| 3  | Design                | 구조, 데이터, 인터페이스, UI 정의              |
| 4  | Implementation        | 언어 선택, 코딩 규칙 정의, 프로그램 작성           |
| 5  | Testing & Integration | 요구 반영 여부, 예상 결과와 유사성 검증, 오류 파악     |
| 6  | Maintenance           | 프로그램 오류 수정, 새로운 요구 추가, 사용 환경 변화 반영 |

---

### 1.3 SDLC 핵심 흐름

    계획
    → 분석
    → 설계
    → 구현
    → 검증
    → 유지보수

시험에서는 SDLC를 “비용과 자원 투입 계획부터 분석, 설계, 구현, 검증, 유지보수에 이르는 전체 개발 프로세스”로 기억한다.

---

## 2. 일반 프로그램과 게임 프로그램의 실행 흐름

### 2.1 일반 프로그램의 실행 흐름

일반적인 배치 프로그램은 수집된 데이터를 특정 시점에 정해진 방식으로 한꺼번에 처리한다.

예:

    데이터 파일
    → 처리 및 가공
    → 정보 파일

---

### 2.2 배치 프로그램

배치 프로그램은 대량의 데이터를 모아 두었다가 특정 시간에 지정된 처리 방법에 따라 일괄적으로 처리하는 프로그램이다.

| 구분    | 설명                          |
|-------|-----------------------------|
| 입력    | 수집된 대량 데이터                  |
| 처리 방식 | 정해진 시간에 일괄 처리               |
| 출력    | 처리된 정보 파일                   |
| 특징    | 사용자의 즉각적 입력보다 정해진 처리 절차가 중요 |

예:

- 은행의 야간 정산
- 대량 로그 분석
- 월별 통계 자료 생성
- 데이터 파일 변환

---

### 2.3 게임 프로그램의 실행 흐름

게임 프로그램은 일반적인 배치 프로그램과 다르다.

게임은 사용자가 버튼을 누르거나 키보드를 입력하거나 마우스를 움직이는 등 사용자의 동작에 따라 실행 흐름이 달라진다.

즉, 게임 프로그램은 실행 흐름을 미리 완전히 예측하기 어렵다.

---

## 3. 이벤트 기반 프로그래밍

### 3.1 이벤트

이벤트는 프로그램에 의해 감지되고 처리될 수 있는 사용자의 동작이나 사건이다.

| 이벤트 예       | 설명               |
|-------------|------------------|
| 키보드 입력      | 사용자가 키를 누름       |
| 마우스 클릭      | 사용자가 버튼이나 화면을 클릭 |
| 타이머         | 일정 시간이 지남        |
| 다른 프로그램의 신호 | 외부 프로그램에서 발생한 사건 |

---

### 3.2 EDP

EDP는 Event Driven Programming의 약자이다.

이벤트 기반 프로그래밍은 실행 흐름을 예측하는 프로그래밍 기법이 무의미한 환경에서 사용된다.

사용자의 입력장치, 타이머, 다른 프로그램 등으로부터 발생하는 이벤트를 중심으로 생각하는 방식이다.

---

### 3.3 이벤트 기반 프로그래밍의 특징

| 구분    | 설명                   |
|-------|----------------------|
| 중심    | 사용자의 동작이나 사건         |
| 실행 흐름 | 이벤트 발생 여부에 따라 달라짐    |
| 주요 활용 | 게임, GUI 프로그램, 모바일 앱  |
| 장점    | 사용자 상호작용에 유연하게 반응 가능 |

---

## 4. EDP 프로그램 구동 구조

### 4.1 EDP의 구성요소

이벤트 기반 프로그램은 다음 요소로 구동된다.

| 구성요소      | 역할                  |
|-----------|---------------------|
| 이벤트 발생 장치 | 이벤트 유형에 따라 이벤트 생성   |
| 이벤트 큐     | 발생한 이벤트를 저장         |
| 이벤트 루프    | 이벤트 발생 여부를 지속적으로 확인 |
| 이벤트 핸들러   | 이벤트 발생에 따른 실행 코드    |

---

### 4.2 EDP 구동 흐름

    이벤트 발생 장치
    → 이벤트 생성
    → 이벤트 큐에 저장
    → 이벤트 루프가 check
    → 이벤트 루프가 get
    → 이벤트 핸들러 실행

---

### 4.3 이벤트 큐와 이벤트 루프

| 개념      | 설명                            |
|---------|-------------------------------|
| 이벤트 큐   | 발생한 이벤트를 순서대로 저장하는 구조         |
| 이벤트 루프  | 이벤트 큐를 계속 검사하고 적절한 이벤트를 꺼내 처리 |
| 이벤트 핸들러 | 이벤트에 대응하여 실제 실행되는 코드          |

시험에서는 이벤트 기반 프로그래밍의 구성요소에 “컴파일러”가 포함되지 않는다는 점을 기억한다.

---

## 5. 게임의 진행과정

### 5.1 게임 루프

게임 루프는 게임이 종료될 때까지 계속 반복되는 핵심 구조이다.

강의의 게임 진행 과정은 다음과 같다.

    게임 초기화
    → 이벤트 검사
    → 이벤트 처리
    → 게임 업데이트
    → 화면 생성
    → 화면 업데이트
    → 게임 종료 조건 확인
    → 종료 전까지 반복

---

### 5.2 게임 루프의 의미

게임 루프는 게임 프로그램이 사용자 입력과 게임 상태 변화에 계속 반응하기 위해 반복 수행하는 구조이다.

| 단계       | 의미                 |
|----------|--------------------|
| 이벤트 검사   | 사용자의 입력이나 사건 확인    |
| 이벤트 처리   | 입력에 따른 동작 처리       |
| 게임 업데이트  | 게임 상태 변경           |
| 화면 생성    | 변경된 상태를 화면에 표현할 준비 |
| 화면 업데이트  | 실제 화면 출력           |
| 종료 조건 확인 | 승리, 무승부, 종료 요청 확인  |

---

## 6. 프로그램 정의 및 설계

## 6.1 프로젝트 이름: 틱택토

이번 실전 프로그래밍 프로젝트는 `틱택토(Tic Tac Toe)` 게임이다.

---

### 6.2 게임 정의

틱택토는 두 명의 플레이어가 3×3 판에 번갈아 가며 O와 X를 기록하는 게임이다.

동일한 표시를 가로, 세로 또는 대각선상에 일직선으로 놓으면 승리한다.

| 항목    | 내용                         |
|-------|----------------------------|
| 게임판   | 3×3                        |
| 플레이어  | 사용자와 컴퓨터                   |
| 기호    | O, X                       |
| 승리 조건 | 같은 기호 3개를 가로, 세로, 대각선으로 연결 |
| 종료 조건 | 승리 또는 빈칸 없음                |

---

### 6.3 게임 규칙

1. 한 플레이어가 아직 기록되지 않은 빈 칸에 자신의 기호를 표시한다.
2. 다른 플레이어가 아직 기록되지 않은 빈 칸에 자신의 기호를 표시한다.
3. 둘 중 한 플레이어가 가로, 세로 또는 대각선을 자신의 기호만으로 채우면 승리한다.
4. 게임은 한 플레이어가 승리하거나 빈 칸이 남지 않아 비길 때까지 계속된다.

---

### 6.4 게임 승리 상황

틱택토의 승리 상황은 전체 8가지이다.

| 구분  | 개수 | 설명                       |
|-----|---:|--------------------------|
| 가로  |  3 | 1행, 2행, 3행               |
| 세로  |  3 | 1열, 2열, 3열               |
| 대각선 |  2 | 왼쪽 위→오른쪽 아래, 오른쪽 위→왼쪽 아래 |
| 합계  |  8 | 전체 승리 경우의 수              |

---

## 7. 틱택토 요구사항

### 7.1 기본 요구사항

| 요구사항    | 설명                      |
|---------|-------------------------|
| 시작 플레이어 | 게임 시작 시 무작위로 선택         |
| 플레이어 구분 | 사용자는 사용자 입력, 컴퓨터는 랜덤 선택 |
| 기호      | 사용자와 컴퓨터가 각각 O와 X 사용    |
| 입력 방식   | 빈 칸의 행, 열 좌표를 입력        |
| 종료 요청   | 사용자가 "00" 입력 시 발생       |
| 출력 방식   | 텍스트 그래픽으로 게임판 출력        |

---

### 7.2 게임 종료 조건

게임은 다음 중 하나가 발생하면 종료된다.

| 종료 조건 | 설명                      |
|-------|-------------------------|
| 승리    | 한 플레이어가 가로, 세로, 대각선을 완성 |
| 무승부   | 빈 칸이 남지 않음              |
| 종료 요청 | 사용자가 "00" 입력            |

---

## 8. 틱택토 설계

## 8.1 틱택토 설계 1/3

### 게임판 생성

게임판은 3×3 크기의 2차원 리스트로 표현한다.

빈칸은 `*` 기호를 사용한다.

    board = [
        ["*", "*", "*"],
        ["*", "*", "*"],
        ["*", "*", "*"]
    ]

---

### 빈칸 확인

게임판에 빈칸이 남아 있는지 확인한다.

    def has_empty_cell(board):
        for row in board:
            for cell in row:
                if cell == "*":
                    return True
        return False

핵심:

- 게임판 전체를 확인한다.
- 빈칸 `*`이 있으면 True를 반환한다.
- 빈칸이 없으면 False를 반환한다.

---

### 승리 여부 확인

승리 상황은 총 8가지이다.

- 각 행 확인
- 각 열 확인
- 두 개의 대각선 확인

  def check_winner(board, mark):
  for i in range(3):
  if board[i][0] == mark and board[i][1] == mark and board[i][2] == mark:
  return True

            if board[0][i] == mark and board[1][i] == mark and board[2][i] == mark:
                return True

        if board[0][0] == mark and board[1][1] == mark and board[2][2] == mark:
            return True

        if board[0][2] == mark and board[1][1] == mark and board[2][0] == mark:
            return True

        return False

---

## 8.2 틱택토 설계 2/3

### 게임판 출력

게임판의 현재 상태를 텍스트 그래픽으로 출력한다.

    def print_board(board):
        for row in board:
            print(row[0], row[1], row[2])

---

### 게임 시작

게임 시작 시 무작위로 선공 플레이어를 선택한다.

    import random

    current_player = random.choice(["user", "computer"])

---

### 게임 루프 1/2

게임판의 현재 상태를 출력하고 다음 플레이어가 빈칸을 선택한다.

| 플레이어 | 선택 방식           |
|------|-----------------|
| 사용자  | 행과 열 번호를 직접 입력  |
| 컴퓨터  | 랜덤으로 행과 열 번호 선택 |

사용자 입력 예:

    user_input = input("행과 열 번호를 입력하세요: ")

    if user_input == "00":
        print("게임 종료 요청")
        exit()

컴퓨터 선택 예:

    row = random.randint(0, 2)
    col = random.randint(0, 2)

---

### 선택 위치에 기호 표시

플레이어가 선택한 위치가 빈칸이면 해당 기호를 표시한다.

    if board[row][col] == "*":
        board[row][col] = mark

---

## 8.3 틱택토 설계 3/3

### 게임 루프 2/2

플레이어가 기호를 표시한 후에는 승리 여부와 무승부 여부를 확인한다.

1. 현재 플레이어가 승리했는지 확인한다.
2. 승리한 경우 메시지를 출력하고 게임 루프를 종료한다.
3. 게임판에 빈칸이 남아 있는지 확인한다.
4. 빈칸이 없으면 무승부 메시지를 출력하고 게임 루프를 종료한다.
5. 게임이 종료될 때까지 게임 루프를 무한 반복한다.

---

### 게임 루프 구조 예

    while True:
        print_board(board)

        # 현재 플레이어가 빈칸을 선택한다.
        # 선택한 위치에 O 또는 X를 표시한다.

        if check_winner(board, mark):
            print(mark, "승리")
            break

        if not has_empty_cell(board):
            print("무승부")
            break

        # 다음 플레이어로 변경한다.

---

## 9. 틱택토 전체 구조 요약

틱택토 프로그램은 이벤트 기반 게임 프로그램의 흐름을 따른다.

    게임 초기화
    → 게임판 출력
    → 선공 플레이어 선택
    → 게임 루프 시작
    → 플레이어 입력 또는 컴퓨터 랜덤 선택
    → 게임판 업데이트
    → 승리 여부 확인
    → 빈칸 여부 확인
    → 화면 업데이트
    → 종료 조건까지 반복

---

## 정리하기

1. 소프트웨어 개발 라이프사이클, SDLC는 비용과 자원 투입 계획부터 분석, 설계, 구현, 검증, 유지보수에 이르는 전체 개발 프로세스를 의미한다.

2. 수집된 데이터를 한 번에 일괄 처리하는 배치 프로그램과 달리, 이벤트 기반 프로그래밍은 실행 흐름 예측이 무의미한 환경에서 사용자의 동작이나 사건을 중심으로 구동된다.

3. EDP의 구동 구조는 발생한 이벤트를 이벤트 큐에 저장하고, 이벤트 루프가 이를 지속적으로 검사하여 알맞은 이벤트 핸들러를 실행하는 형태이다.

4. 틱택토 게임은 3×3 크기의 2차원 리스트를 사용하여 상태를 관리하며, 가로·세로·대각선 방향을 확인하여 총 8가지 승리 상황을 검사하도록 설계한다.

5. 프로그램의 게임 루프는 무작위 선공 결정 후 사용자나 컴퓨터가 입력한 행과 열 번호에 따라 기호 O, X를 표시하고, 빈칸 여부를 확인하여 승패 또는 무승부를 판별하는 과정을 반복한다.

## 시험 직전 체크포인트

### 반드시 암기하기

- SDLC = Planning → Analysis → Design → Implementation → Testing & Integration → Maintenance
- 배치 프로그램 = 대량 데이터 일괄 처리
- 이벤트 = 프로그램이 감지하고 처리할 수 있는 사용자의 동작이나 사건
- EDP = Event Driven Programming
- EDP 구성 = 이벤트 발생 장치, 이벤트 큐, 이벤트 루프, 이벤트 핸들러
- 컴파일러는 EDP 구동 구조 구성요소가 아님
- 게임 루프 = 게임 종료 전까지 반복되는 핵심 구조
- 게임 루프 흐름 = 이벤트 검사 → 이벤트 처리 → 게임 업데이트 → 화면 생성 → 화면 업데이트
- 틱택토 = 3×3 판에 O, X를 번갈아 기록
- 틱택토 승리 조건 = 가로, 세로, 대각선 3칸 일치
- 전체 승리 상황 = 8가지
- 게임판 자료구조 = 3×3 2차원 리스트
- 빈칸 표시 = `*`
- 사용자는 행·열 번호 입력
- 컴퓨터는 랜덤으로 행·열 번호 선택
- 종료 요청 = `"00"`
- 종료 조건 = 승리, 무승부, 종료 요청
